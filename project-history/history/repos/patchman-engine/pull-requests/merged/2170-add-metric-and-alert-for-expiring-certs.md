## Summary by Sourcery

Add certificate expiry monitoring for client certificates and wire it into the metrics subsystem.

New Features:
- Expose a Prometheus gauge reporting days until expiry for monitored client certificates, currently for the Candlepin client cert.
- Introduce a background collector that periodically refreshes certificate expiry metrics based on configured certificate/key material.
- Add a PrometheusRule manifest to alert when monitored client certificates are within 30 days of expiry.

Enhancements:
- Add a shared utility to compute days until expiration from a TLS certificate chain for reuse across components.

Documentation:
- Document the new certificate expiry alerting behavior in the PrometheusRule manifest annotations.

Tests:
- Add unit tests for the certificate expiry metric updater, including handling of invalid and missing certificate configuration.
- Add unit tests for the certificate expiry utility to cover normal, nil, unparsable, and empty certificate chains.

### Comment by @dev-vektr-ai on April 23, 2026 at 02:54 PM UTC

Thanks for the clear metric/alert addition and the focused test coverage. I found two items worth addressing before this moves out of draft:

1. `DaysTillExpiration` silently ignores certificate parse failures and returns `0, nil` when every entry in the chain is unparsable. That can make a broken or malformed cert look like an already-expired/0-day cert instead of surfacing a configuration problem, which may create confusing alerts and hide the real error source.

2. `updateCertificateExpiryGauges()` returns early on `X509KeyPair` / `DaysTillExpiration` errors without updating or clearing the gauge. If the gauge was previously set, Prometheus can keep serving a stale last-good value indefinitely. Consider explicitly resetting/deleting the `candlepin` label (or setting a separate error metric) when the cert/key cannot be parsed so operators do not get a misleading expiry value.

Otherwise the collector wiring and the alert manifest look straightforward.
