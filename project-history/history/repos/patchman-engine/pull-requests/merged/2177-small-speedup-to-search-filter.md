Hey - I've found 2 issues, and left some high level feedback:

- In `ApplySearch`, building the OR conditions off `database.DB` and then passing that into `tx.Where(or)` risks losing scopes/transaction options attached to `tx`; consider starting the OR chain from `tx` (or from a `tx.Session(&gorm.Session{NewDB: true})`) and chaining `Where`/`Or` directly on that instead.
- The new `column::text ILIKE ?` pattern can still inhibit index usage (especially on non-text columns) unless matching functional indexes exist; if index utilization is critical, you may want to avoid `::text` where possible or constrain `searchColumns` to text fields.

<issue_to_address>
**issue (bug_risk):** Building the OR chain off `database.DB` and then passing it into `tx.Where` is likely not doing what you expect and may break transaction scoping.

Because `or` is built from the global `database.DB`, `tx.Where(or)` creates a subquery on the global connection, not on `tx` and its existing predicates. That can drop prior `Where` conditions and break transaction isolation.

Build the OR predicates on `tx` instead, e.g.:

```go
tq := tx
for i, column := range searchColumns {
    cond := column + "::text ILIKE ?"
    if i == 0 {
        tq = tq.Where(cond, "%"+search+"%")
    } else {
        tq = tq.Or(cond, "%"+search+"%")
    }
}
return tq, fmt.Sprintf("search=%s", search)
```

This keeps all filters within the same transaction and avoids using the global handle in query construction.
</issue_to_address>

To make the test more robust, you could, for example:
