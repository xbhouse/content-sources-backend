### Review by @sourcery-ai - Commented on March 24, 2026 at 01:56 PM UTC

Hey - I've found 2 issues, and left some high level feedback:

- The new migration filenames (`148_drop_inventrory_schema.*.sql`) contain a typo in `inventrory` which could make future maintenance/searching harder; consider renaming them to `inventory` before merging.
- In the up migration you wrap `REVOKE cyndi_* FROM ...` in `DO` blocks to tolerate missing roles, but the `REVOKE ALL/USAGE ON SCHEMA inventory FROM cyndi_*` statements are not guarded and will fail if the roles were already dropped; consider wrapping those schema-level REVOKE statements similarly in `DO` blocks or adding IF EXISTS logic.
- The down migration recreates roles and grants, but does not restore any objects inside the `inventory` schema (e.g., views/tables that might have existed); if this is intentional, it might be worth confirming that no consumers rely on those objects being recoverable on rollback.

**issue (bug_risk):** GRANTs to application roles may fail if those roles are missing in some environments.

To keep behavior consistent and more robust across partial/dev setups, consider wrapping these `GRANT`s in the same `DO $$`/`EXCEPTION WHEN undefined_object` pattern.
