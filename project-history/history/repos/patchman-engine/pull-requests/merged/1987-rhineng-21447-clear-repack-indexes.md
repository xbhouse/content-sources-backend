**suggestion (performance):** Consider using DROP INDEX CONCURRENTLY to avoid long locks on large/active tables.

Without `CONCURRENTLY`, these drops take an `ACCESS EXCLUSIVE` lock and can block reads/writes on busy tables. If these indexes are on high-traffic production tables, prefer `DROP INDEX CONCURRENTLY` (noting it must run outside a transaction, so migration tooling may need adjustment). If they’re on small tables or in a maintenance window, the current approach might be fine, but please confirm.

Suggested implementation:

```
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189169";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753183194";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189040";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189283";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753185680";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189153";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189239";

DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753184910";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189118";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189251";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753184976";
DROP INDEX CONCURRENTLY IF EXISTS "public"."index_1753189121";

```

1. Ensure your migration tooling is configured to **run this migration without wrapping it in a single transaction**, because `DROP INDEX CONCURRENTLY` is not allowed inside a transaction block in PostgreSQL. For example, in some frameworks this is done by marking the migration as non-transactional or disabling `transactional` for this file.
2. If your migration runner batches multiple `.sql` files in a single transaction, this file should be excluded from that behavior so each `DROP INDEX CONCURRENTLY` can run in autocommit mode.
