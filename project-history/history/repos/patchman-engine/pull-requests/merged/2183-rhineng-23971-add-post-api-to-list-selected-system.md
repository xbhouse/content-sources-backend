**question (bug_risk):** Preserving existing `filter[id]` query behavior when body `ids` are provided

When `ids` is present in the body, `systemsCommon` always overwrites any existing `filters["id"]` from the query string. For callers using both `filter[id]` and body `ids`, this may produce unexpected results.

Consider either:
- Intersecting existing `filters["id"]` with `ids`, or
- Treating body `ids` as the single source of truth and explicitly rejecting/ignoring conflicting `filter[id]` with a clear error.

Making this interaction explicit in code (e.g., by checking for an existing `filters["id"]`) would reduce the risk of subtle filter bugs.
</issue_to_address>
